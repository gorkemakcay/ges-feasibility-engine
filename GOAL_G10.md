# Goal G10 — Finance Correctness: PySAM Migration & SAM Input-Mapping Validation

> Full specification for autonomous goal G10. The `/goal` condition is a thin pointer to this file.
> Read AGENTS.md, PROJECT_BRIEF.md (§7), and DECISIONS.md before starting. Do ONLY G10.

## Objective
Migrate the financial rollup from hand-rolled Python math to PySAM `Cashloan` + `Utilityrate5`,
so outputs are produced by the same engine as the SAM desktop GUI and become validatable.

## WHY (verified facts — trust these, do not re-derive)
- `finance/engine.py::run_pv_finance` calls PySAM ONLY for LCOE (`PySAM.Lcoefcr` / SimpleLCOE).
  NPV, simple_payback, discounted_payback come from a hand-rolled loop with a hardcoded 0.5% degradation.
- `finance/battery.py::run_pv_storage_finance` does the same; its hand-rolled hourly battery dispatch
  (`_run_dispatch`) is GOOD and must be KEPT — only the financial rollup moves to PySAM.
- `TariffConfig` DEFINES `debt_fraction`, `loan_term`, `loan_rate` but the engines NEVER use them —
  NPV is effectively 100% equity. Fix this (wire them into Cashloan).
- `tests/test_finance.py::test_finance_golden_file` silently REWRITES the golden if it is missing
  (self-healing) — this masks drift. Remove that auto-write so a missing/changed golden fails loudly.
- PySAM IS the same compute engine as the SAM desktop GUI. Correctly-wired PySAM outputs ARE SAM
  outputs; the manual GUI step only verifies input mapping, not the math.

## Scope (ONLY these paths)
`src/gesfeas/finance/`, `tests/`, `tests/golden/`, `config/tariffs/`, `DECISIONS.md`, `VALIDATION.md`.

### 1. `run_pv_finance`
- Value energy via `PySAM.Utilityrate5`: feed the 8760 hourly PV generation (`gen`) and 8760 hourly
  load (`consumption_series`). Map the netting regime to `ur_metering_option` (hourly netting vs
  monthly netting / mahsuplaşma). Accept `netting_mode` as a NEW optional parameter (default = today's
  behavior); do NOT import the regulation module.
- Run `PySAM.Cashloan` (behind-the-meter commercial model) for CAPEX, debt (WIRE UP
  `debt_fraction`/`loan_term`/`loan_rate` so they actually move NPV), OPEX escalation (`inflation_rate`),
  `discount_rate`, `lifetime`, degradation. Take NPV, simple_payback, discounted_payback, and LCOE from
  PySAM outputs — no hand-rolled cash-flow loop may remain.
- Keep pvlib as the production engine; feed `ProductionResult.hourly_production_kwh` as `gen`.
  Do NOT add a PySAM generation model.

### 2. `run_pv_storage_finance`
- Same PySAM rollup, but CRITICAL: KEEP `_run_dispatch`, then feed the POST-DISPATCH net grid series
  (net import/export after battery charge/discharge) into Utilityrate5 — NOT the raw load — so battery
  value flows into the financials.
- `self_consumption_ratio`, `grid_export_ratio`, `battery_cycles_year1` stay computed by the dispatch.
- Battery CAPEX and replacement cost go into Cashloan.

### 3. Interfaces
- Signatures `run_pv_finance(FinanceInput)->FinanceResult` and
  `run_pv_storage_finance(BatteryFinanceInput)->BatteryFinanceResult` unchanged.
- Return the same pydantic model shapes (fields may be added ADDITIVELY only).
- `scenario/engine.py`, `report/`, and `app/` must keep working with NO edits.

### 4. Config-driven
- Move the hardcoded 0.5% degradation to `config/tariffs/*.yaml`.
- Any new knob (metering option mapping, degradation) lives in config or is passed in.
- Currency stays USD — TRY/TOU is explicitly OUT OF SCOPE (separate future goal); do not touch it.

### 5. No silent fallback
- If a PySAM module fails to execute, raise a clear error. Never fall back to hand-rolled math.

### 6. VALIDATION.md
Write a one-time SAM-GUI input-mapping recipe for the G3 reference case:
100 kW; $600/kW CAPEX; $10/kW-yr O&M; 8% discount; 70% debt / 10 yr / 5%; 25 yr life; degradation;
buy $0.12 / sell $0.08; load 100,000 kWh/yr; gen 150,000 kWh/yr flat.
Include: SAM version, financial model = Commercial, every input value in a copy-ready table, and the
expected outputs (NPV, LCOE, simple & discounted payback) from the regenerated golden. State clearly
that goldens are "regenerated by PySAM (same engine as SAM), pending one-time human input-mapping
sign-off in SAM GUI." Include a short "how to sign off" checklist.

## Testing
- Delete and deterministically REGENERATE the affected goldens (`tests/golden/g3_reference.json`,
  `g4_reference.json`, `g9_full_pipeline.json`) from the new PySAM outputs. Intentional change —
  document it in DECISIONS.md.
- Remove the self-healing auto-write in `test_finance.py`; the golden test must FAIL if the file is
  missing or drifts. Tolerances: rel=1e-3 for money metrics, 1e-2 for payback/LCOE.
- **Debt-sensitivity test:** raising `debt_fraction` (or `loan_rate`) changes NPV in the correct
  direction — proves the previously-dead debt params are now live.
- **Netting test:** hourly vs monthly netting on the same gen/load yields different bill savings
  (hourly ≥ monthly for a self-consuming site).
- **Battery-value test:** with the battery present, the net-grid series fed to Utilityrate5 differs
  from raw load, and PV+Storage annual_savings reflects the dispatch (battery adds non-zero incremental
  savings for a night-load / triple-shift case).
- **Config-driven test:** changing degradation in config changes outputs with no code change.
- **Sanity:** doubling system size ≈ doubles CAPEX; absurd CAPEX → NPV < 0; zero consumption behavior
  unchanged.
- ALL non-finance tests (production, regulation, scenario wiring, report, integration) MUST still pass;
  the scenario → report → PDF pipeline MUST run end-to-end on fixture data.
